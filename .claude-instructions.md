# Instructions for Claude Code

**IMPORTANT**: When working on projects in this portfolio, always follow these guidelines.

## Project Context

This is a Technical Product Manager (TPM) portfolio repository. Every project demonstrates:

- Technical understanding and execution
- Product thinking and decision-making
- Professional documentation and communication
- Ability to ship complete, working projects

## Code Quality Standards

### Python

- Use Black formatter (max line length: 100)
- Follow PEP 8 conventions
- Add docstrings to functions and classes
- Use type hints where helpful
- No `print()` statements for debugging (use logging)

### JavaScript/TypeScript

- Use Prettier for formatting
- ES6+ features preferred
- camelCase for variables, PascalCase for classes
- No `console.log()` for debugging (use proper logging)

### General

- Meaningful variable/function names (no `x`, `temp`, `data` without context)
- Small, focused functions (prefer 20 lines or less)
- Comment the WHY, not the WHAT
- Remove all dead/commented code
- Never commit secrets, API keys, or credentials

## Documentation Requirements

Every project MUST have:

1. **README.md** with:

   - Problem statement (what problem does this solve?)
   - Solution approach
   - Tech stack
   - Setup instructions that actually work
   - Usage examples
   - Screenshots or demo
   - Key learnings section

2. **docs/PRD.md** with:

   - User stories
   - Success metrics
   - Requirements (functional and non-functional)

3. **docs/ARCHITECTURE.md** with:
   - System design diagram
   - Key design decisions and rationale
   - Trade-offs considered
   - API documentation (if applicable)

## Development Workflow

1. **Always start with planning**:

   - Fill out PRD.md before writing code
   - Sketch architecture before implementation
   - Define success criteria upfront

2. **Build incrementally**:

   - Start with MVP features
   - Commit frequently with clear messages
   - Update documentation as you build

3. **Focus on completion**:
   - Working end-to-end > partially implemented features
   - Simple and complete > complex and unfinished
   - Professional quality > quantity

## Security & Best Practices

- Validate all user input
- Use environment variables for configuration
- Provide `.env.example` (never commit `.env`)
- Handle errors gracefully
- No hardcoded credentials
- Check for OWASP top 10 vulnerabilities

## TPM-Specific Considerations

Show product thinking by:

- Explaining design decisions in ARCHITECTURE.md
- Documenting trade-offs (speed vs. complexity, cost vs. features)
- Including success metrics in PRD
- Demonstrating user-centric thinking
- Showing prioritization rationale
